### Step 1-1: MusicBrainz Aliases 수집 (~22분)

기존 1,213명 + 신규 200명 아티스트의 **locale별 별칭** 수집.

```python
# /api/v2/artist/{mbid}?inc=aliases&fmt=json
# → aliases[].name, aliases[].locale, aliases[].primary
# Rate limit: 1.1초/요청 → ~1,400명 × 1.1초 ≈ 25분
```

결과: `public/data/artist-aliases.json`
```json
{
  "mbid_abc": {
    "primary": "BTS",
    "aliases": [
      { "name": "방탄소년단", "locale": "ko" },
      { "name": "防弾少年団", "locale": "ja" }
    ]
  }
}
```

### Step 1-2: 메인 표시명 자동 판정 (LLM 0토큰)

```python
def determine_display(mb_aliases):
    ko = next((a for a in aliases if a['locale'] == 'ko'), None)
    primary = next((a for a in aliases if a.get('primary')), None)

    if primary and not has_korean(primary['name']) and ko:
        # BTS, ITZY 케이스 → 영문 메인, 한글은 ALIAS_MAP
        return name=primary['name'], nameKo='', alias=ko['name']

    if primary and has_korean(primary['name']):
        # 아이유, 잔나비 케이스 → 한글 메인
        return name=name, nameKo=primary['name'], alias=[]

    # Fallback: 글자 판정
    return name=name, nameKo=(nameKo if has_korean(nameKo) else '')
```

## 7. 기술 의사결정 기록

### Q: 왜 7 Phase → 4 Phase로 줄였나?

**A**: V7.0.3은 단계가 너무 잘게 쪼개져 있었습니다.
- Phase 1(콜라보)과 Phase 2(이름)는 같은 JSON을 수정하므로 합침
- Phase 3(DB)과 Phase 5(API)와 Phase 6(Admin)은 모두 "오늘의 우주" 기능이므로 합침
- Phase 7(QA)는 Phase 4로 번호 변경

### Q: ALIAS_MAP 병합 vs 덮어쓰기?

**A**: V7.0.3에서는 "자동 생성"이라고만 했지만, 이러면 기존 수동 데이터가 유실됩니다.
- 수동 데이터에는 MB에 없는 닉네임이 많음: "방탄", "블핑", "악뮤", "슈가", "Agust D" 등
- **병합 방식**: MB 자동 데이터를 기존 수동 데이터에 **추가(append)**만 함
- 기존에 이미 매핑이 있는 아티스트는 MB 데이터가 보충만 함

### Q: Vercel CRON 타임아웃 문제?

**A**: Hobby 플랜은 함수 타임아웃 10초.
- 50건 GPT 호출은 불가능 (50 × 1.5초 = 75초)
- **해법**: 1회 CRON에서 **5건만** 처리 (5 × 1.5초 = 7.5초 < 10초)
- 50건이 있으면 10일에 걸쳐 자동 처리됨
- Pro 플랜으로 업그레이드하면 60초 → 1회에 40건 가능

### Q: 발매일 검증을 왜 바로 안 하나?

**A**: MusicBrainz `first-release-date`의 정확도가 이미 ~95%.
- 초기에는 MB 날짜를 그대로 사용 (`verified=false` 상태)
- "오늘의 우주"에서 미검증 앨범도 ⚠️ 마크와 함께 표시
- CRON이 하루 5건씩 점진적으로 검증 → 약 1년이면 전체 완료
- 급하면 Admin에서 수동 트리거

### Q: 콜라보 분리 시 엣지 유실 위험은?

**A**: 백업 + 이관 로그로 대응.
- 분리 전 반드시 `*.bak` 백업 파일 생성
- 이관된 엣지를 `collab_fix_logs`에 기록
- 문제 발견 시 `.bak`에서 복원 가능
