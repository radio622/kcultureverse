## 1. 개요 (Overview)

V7.0.1의 핵심 목표는 두 가지입니다:

1. **자가 발전형 데이터 생태계**: 유저가 자연어로 아티스트 정보와 관계를 제안하고, AI 게이트키퍼가 검증하여 우주 데이터에 반영하는 크라우드소싱 파이프라인 구축
2. **관계 심층 탐구 UX**: 엣지(연결선) 클릭 시 상세 팝업, 두 아티스트 간 관계 경로 검색 등 유저가 K-Culture 관계망을 직관적으로 탐구할 수 있는 인터랙션 추가

### 1.1 V6 → V7.x 달라지는 것과 달라지지 않는 것

| 항목 | V6.9 (현재) | V7.0.1 (계획) |
|------|------------|------------|
| 우주 렌더링 방식 | 정적 JSON + Canvas (`react-force-graph-2d`) | **변경 없음** (5,000+ 노드 시 WebGL 전환 검토) |
| 데이터 파이프라인 | 수동 스크립트 → 빌드 → 배포 | 유저 크라우드소싱 → AI 검증 → 자동 빌드 → 배포 |
| 아티스트 이름/관계 수정 | 개발자가 직접 스크립트 편집 | 유저가 자연어로 요청 → AI가 검증 후 반영 |
| 인증 시스템 | 없음 (퍼블릭) | 구글 로그인 + 프로필 등록 |
| 엣지/관계 탐색 | 노드 클릭 위주 / 우클릭 경로탐색 | **엣지 클릭 팝업 + 듀얼 아티스트 관계 검색 UI 추가** |
| 관계 타입 | 6종 (`SAME_GROUP`, `FEATURED`, `PRODUCER`, `WRITER`, `INDIRECT`, `GENRE_OVERLAP`) | **5종 추가** (COVER 5단계 + LABEL + TV_SHOW) |

## 2. 유저 인증 및 접근 제어 (Authentication & Access Control)

### 2.1 가입 및 인증 흐름

```
[우측 상단 👤 또는 ✏️ 버튼 클릭]
    ↓
[구글 소셜 로그인 (Auth.js v5 + Google OAuth)]
    ↓
[프로필 온보딩 — 필수 입력]
    ├─ 닉네임 (유저 요청 게시판에 표시될 이름)
    ├─ 성별 (선택지: 남 / 여 / 기타 / 밝히지 않음)
    ├─ 연령대 (선택지: 10대 / 20대 / 30대 / 40대 / 50대+)
    └─ 📧 뉴스레터 수신 동의 (체크박스)
    ↓
[프로필 완료 → 가입 완료]
    ├─ 뉴스레터 동의함 → 🌟 **정회원(Full Member)** 승급
    └─ 뉴스레터 미동의 → 🤝 **준회원(Associate Member)**
```

> **준회원 정의**: 구글 로그인은 했으나, **프로필 온보딩을 미완료**했거나 **뉴스레터 수신을 거부**한 사용자.  
> **정회원 승급 조건**: 프로필 완료 + 뉴스레터 수신 동의. 뉴스레터를 나중에 해제하면 정회원 → 준회원 즉시 강등.

- **멤버십 안내 고지 위치**:
  - **회원가입(온보딩) 모달**: "뉴스레터를 수신하시면 **정회원**으로 승급되어, 아티스트 정보 업데이트 요청 권한이 부여됩니다."
  - **마이페이지(정보 수정)**: "현재 준회원입니다. 뉴스레터 수신을 켜면 정회원으로 전환됩니다." / "현재 정회원입니다. 뉴스레터를 끄면 준회원으로 전환되며 에디트 권한이 정지됩니다."
- **로그인 후 UI 보존**: 광활한 우주 탐험을 방해하지 않기 위해, 로그인 상태라도 **우측 상단 모서리에 작고 깔끔한 유저 프로필 아이콘(또는 아바타) 하나만 띄워** 몰입감을 유지합니다. 아이콘 클릭 시 드롭다운 메뉴(마이페이지, 에디트 제안, 자율주행, 로그아웃)가 나옵니다.
- **홈페이지 자연스러운 안내**: 첫 진입 시, 하단에 작게 "가입하면 나만의 우주 비행(자율주행)과 아티스트 추가 요청이 가능해요!" 같은 토스트 알럿을 띄워 자연스럽게 가입을 유도합니다.

### 2.2 권한 및 멤버십 분류

| 역할/등급 | 조건 | 권한 범위 | 데이터 에디트 제안 | 게시판 열람 |
|--------|------|---------|----------------|-----------|
| **Admin** | 관리자 계정 | 무제한 | ⭕ 즉시 반영 | ⭕ 전체 CRUD |
| **정회원** (Full) | 가입 + 뉴스레터 동의 | 🚀 자율주행 가능 | ⭕ 1분 1회 제한 | ⭕ 조회 + 내 요청 수정 |
| **준회원** (Associate) | 가입 + 뉴스레터 미동의 | 🚀 자율주행 가능 | ❌ 불가 | ⭕ 조회만 |
| **비회원** (Guest) | 미로그인 | 기본 탐색 + 음악 재생 | ❌ 불가 | ❌ 접근 불가 |

### 2.3 기술 스택

| 기능 | 기술 |
|------|------|
| 소셜 로그인 | **Auth.js v5** (구 NextAuth.js) + Google OAuth Provider |
| 세션 유지 | JWT 기반 서버리스 세션 (Vercel Edge 호환) |
| 프로필 저장 | Supabase (PostgreSQL) `user_profiles` 테이블 |
| Rate Limit | Next.js API Route 미들웨어 (`X-RateLimit-*` 헤더 기반) |

## 3. AI 게이트키퍼 파이프라인 (AI Gatekeeper Pipeline)

### 3.1 LLM 모델 선정

| 모델 | 대상 | 가격 (2026.03 기준) | 비고 |
|------|------|-------------------|------|
| **Gemini 3.1 Flash-Lite** | 일반 유저 (에디트 제안) | **무료** (Google AI Studio Free Tier) | 앱 내 1분 1회 추가 제한 |
| **GPT-5 Mini** | Admin (빠른 수정) | In $0.80/1M · Out $3.20/1M | 사실검증 없이 파싱만. 월 $2 미만 |
| **Gemini 2.5 Flash-Lite** | Fallback 백업 | 무료 | 3.1 쿼터 소진 시 자동 전환 |

### 3.2 파이프라인 처리 흐름

```
[유저 자연어 입력]
    ↓
[Step 1: LLM → Intent 분류 + 구조화 파싱]
    ├─ NAME_CORRECTION  : 이름 수정
    ├─ EDGE_PROPOSAL    : 관계 추가
    ├─ ARTIST_ADDITION  : 신규 아티스트
    ├─ DATA_CORRECTION  : 기타 데이터 정정
    └─ IRRELEVANT       : 거절
    ↓
[Step 2: 서버 사이드 검증 API 병렬 호출 (Promise.all)]
    ├─ search_musicbrainz()  → MBID / 크레딧 교차 검증 (sleep 1100ms 필수)
    ├─ search_itunes()       → 아티스트/트랙 존재 확인
    └─ (Admin만) search_web() → Google CSE 교차 검증 — 일반 유저는 호출 안 함 (일일 100건 과금 방어)
    ↓
[Step 3: 판정]
    ├─ ✅ APPROVED  : 출처 2건+ 확인 → data_overrides 등록
    ├─ ⏳ PENDING   : 검증 모호 → 로그 기록
    └─ ❌ REJECTED  : 허위/무관 → 거절
    ↓
[Step 4: Supabase universe_edit_logs 기록]
```

> ⚠️ **설계 핵심**: LLM에게 Tool(Function Calling)을 위임하지 않고, 서버 코드에서 직접 API를 호출하는 **하이브리드 방식**. Flash-Lite의 Tool 지원 여부에 의존하지 않으며, 검증 로직이 서버에 있으므로 테스트 가능하고 안정적입니다.

## 4. 데이터베이스 설계 (Database Schema)

```sql
-- 유저 프로필
CREATE TABLE user_profiles (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  google_id     TEXT UNIQUE NOT NULL,
  email         TEXT NOT NULL,
  nickname      TEXT UNIQUE NOT NULL,
  gender        TEXT CHECK (gender IN ('male','female','other','undisclosed')),
  age_group     TEXT CHECK (age_group IN ('10s','20s','30s','40s','50s+')),
  newsletter    BOOLEAN DEFAULT false,
  role          TEXT DEFAULT 'user' CHECK (role IN ('admin','user')),
  membership    TEXT DEFAULT 'associate' CHECK (membership IN ('associate','full')),
  created_at    TIMESTAMPTZ DEFAULT NOW()
);

-- 유저 제안 로그 및 AI 판정 기록
CREATE TABLE universe_edit_logs (
  id            SERIAL PRIMARY KEY,
  user_id       UUID REFERENCES user_profiles(id) ON DELETE SET NULL,
  intent        TEXT NOT NULL,
  raw_input     TEXT NOT NULL,
  parsed_data   JSONB,
  status        TEXT DEFAULT 'pending',
  ai_reasoning  TEXT,
  ai_sources    JSONB,
  applied_at    TIMESTAMPTZ,
  created_at    TIMESTAMPTZ DEFAULT NOW()
);

-- 승인된 오버라이드 패치
CREATE TABLE data_overrides (
  id            SERIAL PRIMARY KEY,
  edit_log_id   INT REFERENCES universe_edit_logs(id),
  target_type   TEXT NOT NULL,  -- 'node_name', 'edge', 'node_add', etc.
  target_id     TEXT,
  patch_data    JSONB NOT NULL,
  applied       BOOLEAN DEFAULT false,
  created_at    TIMESTAMPTZ DEFAULT NOW()
);

-- 자율주행 비행 기록
CREATE TABLE flight_logs (
  id            SERIAL PRIMARY KEY,
  user_id       UUID REFERENCES user_profiles(id) ON DELETE CASCADE,
  start_artist  TEXT NOT NULL,
  path          JSONB NOT NULL,
  total_stops   INT NOT NULL DEFAULT 0,
  total_seconds INT NOT NULL DEFAULT 0,
  created_at    TIMESTAMPTZ DEFAULT NOW()
);
```

## 5. 관계 모델 및 가중치 확장 (Expanded Relationship Models)

### 5.1 커버곡(Cover/Remake) 가중치 세분화

기존 `COVER` / `INDIRECT` 관계를 5단계로 세분화합니다. LLM이 유저 피드백 원문을 파싱하여 자동 분류하고, 해당 가중치를 `data_overrides`로 반영합니다.

| 서브타입 | 가중치 | 기준 | 예시 |
|---------|--------|------|------|
| `COVER_OFFICIAL` | **0.7** | 공식 음원 발매 리메이크 (스트리밍 플랫폼 등록) | 아이유 '꽃갈피' 시리즈 |
| `COVER_FULL` | **0.5** | 방송/SNS에서 풀(Full) 버전 커버 (음원 미발매) | '유희열의 스케치북' 풀 커버 |
| `COVER_PARTIAL` | **0.3** | 방송/SNS에서 일부분만 커버 | 라디오 코너에서 1절만 |
| `COVER_UNCLEAR` | **0.2** | 커버했으나 분량 불분명 | 팬캠/짧은 영상 등 |
| `COVER_HUMMING` | **0.1** | 즉흥적 흥얼거림 수준 | 인스타 라이브에서 코러스만 |

**데이터 수집 경로**: 유저가 피드백 게시판에서 "XXX가 YYY의 곡 ZZZ를 유스케에서 풀 커버했어요"라고 자연어 입력 → LLM이 `{ type: "EDGE_PROPOSAL", subtype: "COVER_FULL", weight: 0.5 }`로 파싱 → 서버 검증 → 반영.

### 5.2 코드 반영: V5EdgeRelation 타입 확장 필요

현재 `src/lib/graph-v5.ts`의 `V5EdgeRelation` 타입:
```typescript
// 현재
export type V5EdgeRelation =
  | "SAME_GROUP" | "FEATURED" | "PRODUCER" | "WRITER" | "INDIRECT" | "GENRE_OVERLAP";

// V7.0.1 확장
export type V5EdgeRelation =
  | "SAME_GROUP" | "FEATURED" | "PRODUCER" | "WRITER" | "INDIRECT" | "GENRE_OVERLAP"
  | "COVER_OFFICIAL" | "COVER_FULL" | "COVER_PARTIAL" | "COVER_UNCLEAR" | "COVER_HUMMING"
  | "LABEL" | "TV_SHOW";
```

`GraphCosmos.tsx`의 `EDGE_COLORS` 및 `EDGE_WIDTH`에도 신규 관계 색상/두께를 등록해야 합니다.

### 5.3 기타 간접 관계

| 타입 | 가중치 | 기준 |
|------|--------|------|
| **INDIRECT** | 0.1 / 0.2 / 0.3 | 비-노드 인물(작사가·프로듀서) 공유 곡 수 (1/2/3+) |
| **LABEL** | 0.1 ~ 0.2 | 같은 기획사/레이블 소속 (동시기 0.2, 다른 시기 0.1) |
| **TV_SHOW** | 0.1 ~ 0.3 | 방송 공동 출연 (같은 프로·다른 시간대 0.1, 1회 게스트 0.2, 2회+ 고정 0.3) |

## 6. UX 개선: 관계(Edge) 심층 탐색 기능

### 6.1 엣지 클릭 시 관계 상세조회 팝업

**기술 검증 ✅**: `react-force-graph-2d`는 `onLinkClick` prop을 지원하며, 현재 `GraphCosmos.tsx`에서 사용하고 있지 않으므로 추가만 하면 됩니다.

- **구현 방식**: `ForceGraph2D`에 `onLinkClick` 핸들러를 추가하여 클릭된 엣지의 `source`, `target`, `relation`, `label`, `weight` 정보를 읽고, 클릭 좌표에 반투명 팝업을 렌더링합니다.
- **팝업 내용 예시**:
  ```
  아이유 ↔ 가인
  ─────────────────
  작사가 김이나 3곡 겹침
    (좋은 날, 너랑 나, 분홍신)
  프로듀서 테디 12곡 겹침
    (...)
  방송 동시 출연 1회
    (유희열의 스케치북)
  ─────────────────
  총 가중치: 0.65
  ```
- **UX 보호 장치**:
  - 엣지의 **정중앙/몸통 부분** 근처를 클릭할 때만 트리거 (가장자리 오클릭 방지)
  - `onLinkClick` 핸들러에서 클릭 좌표와 엣지 중심점 간 거리를 계산하여, 양 끝 20% 범위 내의 클릭은 무시
  - 팝업은 **반투명** glassmorphism 스타일, 다른 곳 클릭 시 자동 dismiss
  - 정보가 많을 경우 (예: 공유 곡 10개+) 최대 3건만 노출 + "외 N건 더보기" 접이식

### 6.2 듀얼 아티스트 관계 검색 (A ↔ B Relationship Search)

**기술 검증 ✅**: `GraphCosmos.tsx`에 이미 **Dijkstra 경로탐색** 함수와 **adjList**(인접 리스트)가 구현되어 있습니다(우클릭 Pathfinding). 이 인프라를 확장하여 UI를 추가합니다.

- **검색 UI 위치**: 기존 `FloatingSearch` 컴포넌트를 확장하여, 검색 패널 상단에 **[단일 검색] / [관계 검색]** 탭 전환 UI를 추가합니다.
- **관계 검색 입력**: 두 개의 입력 필드 (아티스트 A, 아티스트 B)
- **결과 표시**:
  1. **직접 연결**: A↔B 간 직접 엣지가 있으면 가중치와 관계 타입 표시
  2. **간접 연결**: 직접 엣지가 없으면 Dijkstra 최단 경로를 탐색하여 중간 아티스트 경유지를 자연어로 설명
     - 예: "[아이유]는 (작사가 김이나 3곡 공유) [가인]과, [가인]은 (같은 레이블 AOMG) [박재범]으로 연결됩니다"
  3. **카메라 애니메이션**: 기존 `fg.centerAt()` + `fg.zoom()` API를 활용하여 경로를 따라 순차적으로 패닝
- **UI 설계**: 검색결과 창은 우주의 별자리를 가리지 않도록 화면 모서리에 **반투명 플로팅 패널**로 배치. 별과 선이 패널 뒤로 보이도록 `backdrop-filter: blur(12px)` + `background: rgba(7,9,18,0.88)` 처리.

### 6.3 10,000 노드 시 UX 보호

| 우려 사항 | 대응 |
|----------|------|
| 엣지가 너무 많아 클릭 타겟 식별 불가 | 줌 레벨 `< 0.6`에서는 `onLinkClick` 비활성화 (현재 `linkCanvasObject`도 동일 기준 사용) |
| 관계 검색 시 Dijkstra 성능 | 10,000 노드 + 60,000 엣지에서 최악 O(V²) ≈ 1억. **우선순위 큐(Min-Heap)** 적용 시 O((V+E)·logV) ≈ 수십만, <50ms 소요 |
| 팝업이 노드/엣지를 가림 | 팝업은 항상 클릭점에서 약간 오프셋, 투명 배경, ESC/외부클릭으로 즉시 사라짐 |

## 7. 확장성: 10,000명 아티스트 노드 대응 전략 (Scalability Plan)

### 7.1 현재 규모와 예상 성장

| 항목 | V6.9 현재 | V7.x (1년 후) | 장기 (2년 후) |
|------|----------|-------------|-------------|
| 아티스트 (노드) | **1,213명** | 2,500~4,000명 | 5,000~10,000명 |
| 연결선 (엣지) | **3,196개** | 8,000~15,000개 | 30,000~60,000개 |
| v5-layout.json | **151KB** | ~350KB | ~800KB |
| v5-edges.json | **511KB** | ~1.2MB | ~3MB |
| d3-force 빌드 시간 | ~45초 | ~2분 | ~10분 |

### 7.2 단계별 해법

| 문제 | 트리거 시점 | 해결 방안 |
|------|-----------|----------|
| layout.json > 500KB | ~3,000명 | gzip 전송(Vercel 기본)으로 실질 전송량 80% 감소 |
| edges.json > 3MB | ~5,000명 | **Spatial Chunking**: 현재 뷰포트 근처 엣지만 지연 로딩 |
| Canvas FPS 저하 | ~8,000명 | **WebGL 기반 렌더러** 전환 검토 (`react-force-graph-3d` 2D 모드 또는 `pixi-viewport`) |
| d3-force 빌드 > 5분 | ~5,000명 | **증분 빌드**: 기존 좌표 유지 + 신규 노드만 추가 시뮬레이션 (빌드 10분→30초) |

### 7.3 Dijkstra 성능 개선 (Planned)

현재 `GraphCosmos.tsx`의 Dijkstra 구현은 단순 `Set` 기반 O(V²)입니다. 10,000 노드에서는 **Min-Heap (Binary Heap)** 기반으로 교체하여 O((V+E)·logV)로 최적화합니다. 이는 듀얼 아티스트 관계 검색과 자율주행 경로 탐색 모두에 적용됩니다.

## 12. 2026년 기준 API 차단 및 과금 리스크 분석 (High Priority)

최근 2025~2026년에 걸쳐 주요 음악 API들의 보안 및 과금 제약이 매우 타이트해졌습니다. 이에 따라 우주가 중단되는 사태를 막기 위해 다음 사항들을 아키텍처에 강제합니다.

### 12.1 외부 API 위험도 및 방어 아키텍처

| 대상 API | 2026년 주요 제약 (Risk) | 당면 위험 | 방어 및 우회 아키텍처 (Mitigation) |
|---------|--------------------|----------|--------------------------------|
| **Spotify API** | `Related Artists`, `Recommendations` Endpoint **강제 삭제** (2024.11). Free Tier `preview_url` 응답 대부분 제거 | 기능 작동 불능 (자율주행 시 이웃 찾기 및 미리듣기 불가) | ✅ **자체 해결됨**: Spotify API 대신, 이미 구축된 우주의 `adjList`(인접 리스트)를 네비게이션 엔진으로 사용하고, 오디오는 **iTunes API**로 완전히 대체 |
| **MusicBrainz** | 강력한 IP Rate Limit (초당 1건). 위반 시 HTTP 503 및 영구 IP Block | 유저 여러 명이 동시에 에디트 제출 시 서버리스 IP 차단 | ✅ AI 검증 호출 시 **의도적 딜레이(await sleep)** 적용 및 503 발생 시 **Exponential Backoff(백오프) 재시도 로직 필수** |
| **iTunes Search** | 미인증 API. 단일 IP 기준 대량 호출 시(분당 20건+) 차단됨 | 릴레이 미리듣기를 위해 서버가 큐를 가져올 경우 Vercel 서버 IP 영구 차단 | ✅ 자율주행/릴레이 오디오용 iTunes API 호출은 절대 서버(API Route)에서 하지 않고, **유저 브라우저(Client-Side fetch)에서 직접 호출**하도록 분산 설계 |
| **Genius API** | 가사 데이터 스크래핑 전면 차단 (Cloudflare 봇 방어 강화) | 연결 지속 시 서버 IP 차단 | ✅ V7.0.1 AI Gatekeeper 팩트체크 도구에서 **Genius 사용 완전히 배제** (MusicBrainz + iTunes 조합만 사용) |
| **Google CSE** | 일일 무료 허용량 단 **100건**. 1,000건당 $5 부과 | 일반 유저들이 팩트체크를 남발하면 당일 몇 시간 만에 과금 발생 | ✅ 일반 유저의 `search_web()`은 **비활성화**하거나 백업용으로만 분리. MBID/iTunes만으로 1차 검증하고 LLM 자체 지식 활용 |
| **Gemini Flash** | Google AI Studio 무료 티어: 하루 1,500건, 분당 15건 제한 | 바이럴 타서 요청 폭주 시 429 Too Many Requests 에러 발생 | ✅ 인앱 단위로 유저당 **1분 1회 Rate Limit** 하드코딩 + Vercel AI SDK Fallback `[gemini-3.1-flash, gemini-2.5-flash]` 이중화 |
