## 2. 콜라보레이션 노드 수정 전략 (핵심!)

### 2.1 문제 정의

현재 우주에는 "아티스트A;아티스트B" 형태의 **잘못된 단일 노드**가 85건 존재합니다.  
이들은 실제로는 **2명 이상의 개별 아티스트가 정식 협업한 곡**을 뜻하며, 별도의 아티스트가 아닙니다.

**잘못된 상태 (현재):**
```
[Crush;태연] ← 하나의 노드로 존재 (❌ 오류!)
```

**올바른 상태 (수정 후):**
```
[Crush] ──── FEATURED (0.7) ────── [태연]
   ↑ 각각 독립 노드               ↑ 각각 독립 노드
   (기존에 이미 있을 수 있음)       (기존에 이미 있을 수 있음)
```

### 2.2 자동 수정 알고리즘

```python
for each collab_node in layout.nodes:
    if ';' in collab_node.name:
        parts = collab_node.name.split(';')
        
        for each artist in parts:
            # 1. 기존 우주에 해당 아티스트 개별 노드가 있는지 확인
            existing = find_node_by_name(artist)
            if not existing:
                # 2. 없으면 새 노드 생성 (콜라보 노드 좌표 근처에 배치)
                create_node(artist, x=collab_node.x ± random, y=collab_node.y ± random)
        
        # 3. 모든 파트 아티스트 간 FEATURED 엣지 생성 (가중치 0.7)
        for pair in combinations(parts, 2):
            add_edge(pair[0], pair[1], relation="FEATURED", weight=0.7,
                     label="정식 협업 음원 발매")
        
        # 4. 콜라보 노드에 연결된 기존 엣지 → 개별 아티스트로 재연결
        for edge in edges_connected_to(collab_node):
            redirect_to_first_artist(edge)  # 첫 번째 아티스트에게 이관
        
        # 5. 콜라보 노드 삭제
        delete_node(collab_node)
```

### 2.3 수정 시 주의사항

| 상황 | 처리 |
|------|------|
| `Crush;태연` → Crush, 태연 모두 기존 우주에 존재 | 엣지만 추가, 노드 생성 불필요 |
| `10CM;이수현` → 10CM 존재, 이수현 미존재 | 이수현 노드 생성 + 엣지 추가 |
| `김경호;김종서;윤도현;박완규;...` (5명+) | 모든 쌍에 FEATURED 엣지 추가 (조합), 또는 첫 번째 아티스트 중심 star topology |
| 콜라보 노드에 기존 엣지 연결 | 첫 번째 아티스트에게 이관 (모든 기존 관계 보존) |
| 외국 아티스트 포함 (`Crowd Lu;검정치마`) | 한국 아티스트만 노드 생성, 외국 아티스트는 label에 기록 |

### 2.4 콜라보 → 엣지 가중치 규칙

정식 협업 음원 발매는 매우 강한 관계입니다.

| 콜라보 유형 | relation | weight | 기준 |
|------------|----------|--------|------|
| 정식 음원 콜라보 (싱글/앨범 공동 발매) | `FEATURED` | **0.7** | 공식 음원 발매 |
| OST 듀엣 | `FEATURED` | **0.6** | OST 사운드트랙 협업 |
| 라이브/이벤트 합작곡 | `FEATURED` | **0.4** | 비정식 합작 |

## 3. 한글/영어 이름 통합 전략 (자동화)

### 3.1 이름의 3계층 시스템

화면 표시와 검색은 세 가지 레이어로 작동합니다:

| 계층 | 필드 | 역할 |
|------|------|------|
| **`name`** | 영문 공식명 | 노드 ID 역할 + 영문 검색 |
| **`nameKo`** | 한글 공식명 | **화면 표시 메인** + 한글 검색 |
| **`ALIAS_MAP`** | 추가 별칭 | **검색 전용** (화면 표시 안 됨) — 약칭/외래어/본명 등 |

화면 표시 로직: `nameKo || name` → nameKo가 있으면 한글, 없으면 영문이 메인.

### 3.2 메인 표시명 자동 판정 — MusicBrainz Aliases (LLM 불필요!)

MusicBrainz는 아티스트별로 **locale 태그가 달린 별칭(alias)** 데이터를 제공합니다:

```json
{
  "aliases": [
    { "name": "방탄소년단", "locale": "ko", "type": "Artist name" },
    { "name": "BTS", "locale": null, "type": "Artist name", "primary": true },
    { "name": "防弾少年団", "locale": "ja", "type": "Artist name" }
  ]
}
```

이 데이터로 **LLM 토큰 0개**로 메인 표시명을 결정합니다:

```python
def determine_display_name(name, nameKo, mb_aliases):
    """
    3단계 자동 판정 (LLM 비용 = $0)
    """
    ko_alias = next((a for a in mb_aliases if a['locale'] == 'ko'), None)
    primary  = next((a for a in mb_aliases if a.get('primary')), None)
    
    # Step 1: MB primary가 영문이고, ko locale 별칭 존재
    #   → 영문이 국제 공식명 (BTS, ITZY 등)
    #   → nameKo 비움 → 영문이 화면 메인
    #   → ko_alias → ALIAS_MAP에 등록 (한글 검색 보장)
    if primary and not re.search(r'[가-힣]', primary['name']) and ko_alias:
        return {
            'name': primary['name'],      # "BTS" → 화면 메인
            'nameKo': '',                  # 비움 → name이 화면에 표시
            'aliases': [ko_alias['name']], # "방탄소년단" → ALIAS_MAP
        }
    
    # Step 2: MB primary가 한글이면 → 한글이 메인 (아이유, 잔나비)
    if primary and re.search(r'[가-힣]', primary['name']):
        return {
            'name': name,                  # 영문명 유지
            'nameKo': primary['name'],     # "아이유" → 화면 메인
            'aliases': [],                 # 이미 name으로 영문 검색 가능
        }
    
    # Step 3: MB 정보 없으면 글자 판정
    if re.search(r'[가-힣]', nameKo or name):
        return {'name': name, 'nameKo': nameKo or name, 'aliases': []}
    
    return {'name': name, 'nameKo': '', 'aliases': []}  # 영문 메인
```

### 3.3 판정 예시

| 아티스트 | MB primary | MB ko locale | 판정 결과 | 화면 표시 | 한글 검색 |
|---------|-----------|-------------|----------|---------|----------|
| BTS | `"BTS"` (en) | `"방탄소년단"` | 영문 메인 | **BTS** | "방탄소년단" via ALIAS_MAP ✅ |
| 아이유 | `"아이유"` (ko) | - | 한글 메인 | **아이유** | nameKo ✅ |
| ITZY | `"ITZY"` (en) | `"있지"` | 영문 메인 | **ITZY** | "있지" via ALIAS_MAP ✅ |
| 잔나비 | `"잔나비"` (ko) | - | 한글 메인 | **잔나비** | nameKo ✅ |
| 검정치마 | `"검정치마"` (ko) | - | 한글 메인 | **검정치마** | nameKo ✅ |

### 3.4 비용 분석

| 단계 | 비용 | 정확도 | 비고 |
|------|------|--------|------|
| MusicBrainz aliases API | **$0** | ~97% | 추가 API 1회/아티스트 (rate limit 1.1초) |
| 한글 포함 regex fallback | **$0** | ~90% | MB 미발견 아티스트용 |
| ~~LLM 판정~~ | ~~비쌈~~ | - | **불필요** |

### 3.5 검색 양방향 지원 — ALIAS_MAP 자동 생성

`ALIAS_MAP`을 수동으로 하드코딩하는 대신, MB aliases 데이터에서 **자동 생성**합니다:

```python
# 빌드 타임에 자동 생성
for artist in all_artists:
    for alias in artist.mb_aliases:
        if alias['locale'] == 'ko' and alias['name'] != artist.nameKo:
            ALIAS_MAP[artist.name].append(alias['name'])
        if alias['locale'] in (None, 'en') and alias['name'] != artist.name:
            ALIAS_MAP[artist.nameKo].append(alias['name'])
```

### 3.6 한글 이름 보강 작업 (427명)

현재 영어만 설정된 427명 한글명 보강 방법:

1. **자동 (97%)**: MusicBrainz `aliases` API → `locale: "ko"` 별칭 추출
2. **Fallback**: 이름에 한글이 포함되어 있으면 그대로 사용
3. **수동**: MB에도 한글 별칭이 없는 경우 → Admin에서 직접 수정

## 5. "오늘의 우주" 기능 (Today's Universe)

### 5.1 핵심 흐름

```
매일 새벽 2:00 (CRON):
  album_releases에서 오늘+내일 발매 앨범 조회
  → GPT-5 Nano 교차 검증 (MusicBrainz × 나무위키 × 위키피디아 × 네이버)
  → 틀린 날짜 수정 + 검증 완료 마킹
  → daily_verification_logs 기록

유저 런타임:
  GET /api/universe/today-albums?month=3&day=22
  → verified=true 앨범만 반환
  → 자율주행 출발점으로 활용

Admin:
  /admin/calendar → 캘린더형 발매일 뷰 + 수동 정정
```

## 8. 기술 의사결정 기록

### Q: 왜 콜라보 수정이 "최우선"인가?

**A**: 잘못된 노드가 85건이나 있으면:
- Dijkstra 경로탐색 결과가 왜곡됨 (실제 아티스트가 아닌 유령 노드를 경유)
- 자율주행이 유령 노드에 도달하면 미리듣기 실패
- 유저가 "Crush;태연"이라는 존재하지 않는 아티스트를 발견하면 데이터 신뢰도 추락
- 정정 과정에서 FEATURED 엣지가 추가되므로 **관계 데이터가 오히려 풍부해짐**

### Q: Spotify 발매일을 왜 안 쓰나?

**A**: Spotify는 재발매/리마스터 시 원본 발매일을 덮어써버리는 경우가 많습니다.
- 예: 이문세의 1985년 앨범이 Spotify에서 `release_date: 2020-01-15`로 표시
- MusicBrainz의 `first-release-date`가 원본 발매일에 가장 정확
- GPT-5 Nano로 나무위키/위키피디아 교차 확인하여 이중 검증

### Q: 5명 이상 콜라보는 어떻게 처리?

**A**: Star topology 방식을 사용합니다.
- `김경호;김종서;윤도현;박완규;...` (7명) → 모든 N(N-1)/2 쌍에 엣지를 추가하면 21개나 되므로 비효율
- 대신 **첫 번째 아티스트(대표)**를 중심으로 나머지와 1:1 엣지 생성 (6개)
- 제목에 명시된 대표 아티스트가 있으면 그를 허브로 사용

### Q: 한글/영문 메인 표시명을 어떻게 판정하나?

**A**: MusicBrainz aliases 기반 3단계 자동 판정 (**LLM 비용 = $0**):
1. MB `primary` alias가 영문 + `locale: "ko"` 별칭 존재 → 영문 메인, 한글은 ALIAS_MAP
2. MB `primary` alias가 한글 → 한글 메인
3. MB 없으면 → 이름에 한글 포함 여부로 판정 (regex)

나무위키/네이버/멜론 등의 한국 사이트에서의 표기와 일치하는 결과를 위해:
- MusicBrainz의 `primary` = 국제적으로 가장 널리 쓰이는 이름
- MusicBrainz의 `locale: "ko"` = 한국 사이트에서 주로 쓰이는 한글명
- 이 조합만으로 ~97% 정확도 달성. LLM 불필요.
