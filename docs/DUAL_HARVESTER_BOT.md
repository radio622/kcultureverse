## 1. 배경: 왜 듀얼봇이 필요한가?

### 기존 데이터 소스의 한계 (2026-03-23 논의에서 실증)

| 소스 | 장점 | 치명적 한계 |
|------|------|-------------|
| **MusicBrainz** | 일부 K-Pop 대형 아이돌 데이터 완벽 | 인디/발라드 아티스트(예: 백아) 앨범 **아예 없음** |
| **iTunes API** | 한국 인디까지 99.9% 커버 | 한글 앨범명 대신 **영문 번역명 반환** (예: '우주선' → 'SPACESHIP') |
| **Spotify API** | **완벽한 한글 앨범명** 반환 확인 완료 | 발매일 부정확 + 곡별 크레딧 미제공 |

### 결론
- 껍데기(앨범명, 커버, 트랙 수) → **Spotify**
- 발매일 교정 + 크레딧 추출 → **Gemini 3.1 실시간 웹 검색으로 보완**
- 두 개를 결합한 **듀얼 봇(2인 1조)** 구조가 탄생

## 2. 봇 아키텍처

### 전체 작동 흐름도
```
[메인 루프] ─── while True (무한 순환) ───
  │
  ├─ 스케줄러: Supabase에서 엣지 많은 아티스트 순으로 큐 생성
  │           (이미 수집 완료한 아티스트는 자동 건너뜀)
  │
  ├─ 아티스트 1명 할당
  │
  ├─ 🤖 봇 1 (The Harvester) ── Spotify API
  │   ├─ 아티스트 이름 검색 (1회 호출)
  │   ├─ 앨범 10개씩 페이징 수집 (limit=10)
  │   │   └─ 다음 페이지 있으면 → 32초 쿨타임 후 이어서
  │   ├─ 날짜 형식 보정: "1986" → "1986-01-01", "2024-03" → "2024-03-01"
  │   ├─ 수집 데이터: 한글 앨범명, 발매일, 커버URL, 타입, 트랙수
  │   ├─ Supabase album_releases에 upsert (중복 자동 무시)
  │   └─ progress.json에 "수집 완료" 체크포인트 기록
  │
  ├─ 🧠 봇 2 (The Curator) ── Gemini 모델 폴백 체인
  │   ├─ 봇 1이 가져온 앨범 중 미검증 건만 필터링
  │   │   └─ 이중 체크: progress.json + Supabase verified 필드
  │   ├─ 📅 달력 우선순위 정렬:
  │   │   ├─ 1순위: 오늘(월/일)과 같은 발매일 앨범
  │   │   ├─ 2순위: 내일 → 모레 → ... 가까운 날짜 순
  │   │   └─ 3순위: 해당 없는 나머지 미검증 앨범
  │   ├─ 앨범 1건씩 Gemini에게 전달
  │   │   └─ 모델 폴백: 3.1(+search) → 3.1(no-search) → 2.5(+search) → 2.5(no-search)
  │   ├─ -01로 끝나는 의심 날짜에 대한 강화 프롬프트 자동 삽입
  │   ├─ 검증 3대 임무:
  │   │   ├─ ① 최초 한국 실물 발매일 교정
  │   │   ├─ ② 수록곡별 작사/작곡/프로듀싱/피처링 크레딧 추출
  │   │   └─ ③ 아티스트 국적(Korean 여부) 확인
  │   ├─ 검증 결과 → Supabase PATCH 즉시 반영 (라이브 사이트 실시간 업데이트)
  │   ├─ 검증 후 32초 쿨타임
  │   └─ 성공 시만 progress.json에 "검증 완료" 깃발 (실패 시 깃발 안 꽂아 다음 가동 때 재시도)
  │
  ├─ ⏳ 10분 대기 후 다음 아티스트
  │
  └─ 전체 1사이클 완료 시 → 진행률 초기화 → 다시 처음부터 (신규 앨범 체크)
```

### 재처리 방지 3중 잠금 (Triple Dedup Guard)

| 방어선 | 방어 대상 | 기록 위치 | 동작 |
|--------|----------|----------|------|
| 🔒 봇 1 잠금 | 아티스트 재수집 | `progress.json → harvested_artists[]` | 이미 수집한 아티스트는 큐에서 제외 |
| 🔒 봇 2 로컬 잠금 | 앨범 재검증 | `progress.json → verified_albums[]` | `아티스트명::앨범명` 키로 중복 체크 |
| 🔒 봇 2 DB 잠금 | 앨범 재검증 | `Supabase verified=true` | progress.json 유실 시에도 DB에서 확인 |
| 🔒 DB 잠금 | 앨범 중복 Insert | `Supabase ON CONFLICT` | 같은 (artist_name, album_title) 조합 자동 무시 |

### 중단 후 이어서 하기 (체크포인트 복원)
- 스크립트 중단 시 `progress.json`이 자동 저장됨
- 재시작 시 파일을 읽고 마지막 처리 지점 다음부터 이어서 작업
- progress.json 유실 시에도 Supabase의 `verified=true` 필드로 이중 복원
- 전체 아티스트 1바퀴 완료 시 `harvested_artists` 초기화 → 재순환

## 3. API 기술 실증 결과 (2026-03-23)

### Spotify API
- **토큰 발급**: `client_credentials` 방식 → 코드에서 자동 발급 (사이클마다 갱신)
- **한글 앨범명**: `백아 → '우주선'`, `아이유 → 'LILAC'` 등 완벽 한글 출력 확인
- **페이지 한도**: `limit=10` 확정 (50, 20 모두 400 에러. 잠수함 패치 2026-03 기준)
- **아이유 전체 수집**: 9페이지 → 74개 앨범 100% 수집 성공
- **날짜 형식 주의**: 옛날 앨범은 "1986" (연도만) 반환 → 코드에서 "1986-01-01" 자동 보정

### Gemini API
- **모델명 주의**: `gemini-3.1-flash-lite` ❌ (404) → `gemini-3.1-flash-lite-preview` ✅
- **백업 모델**: `gemini-2.5-flash-lite` ✅ (쿼타 독립 확인)
- **실시간 웹 검색**: `"tools": [{"google_search": {}}]` — 3.1에서 확인, 2.5는 400 에러 시 도구 없이 재시도
- **쿼타 구조**: 모델별 독립 관리 (3.1 소진 시 2.5는 멀쩡)

## 4. 실행 방법

### 가동 명령어
```bash
# 맥북 잠자기 방지 + 실행
caffeinate -i python3 scripts/v7.5-dual-harvester-draft.py &

# 실시간 로그 모니터링 (별도 터미널)

kill $(jobs -p)
```

### 진행 상황 확인
- **로그 파일**: `logs/harvester_YYYYMMDD.log` (날짜별 자동 생성)

## 6. 데이터베이스 용량 모니터링

### Supabase (앨범 텍스트 데이터)
- **무료 한도**: 500MB → 앨범 10만 건 = ~20MB. **걱정 불필요**
- **이미지**: 스포티파이 서버 핫링크 → 0 Byte

### Neo4j AuraDB (그래프 엣지/노드)
- **무료 한도**: 관계선 **최대 175,000개** ⚠️
- **현재**: ~8,400개 (여유)
- **대비책**: 메인 크레딧만 엣지 생성 / 12만 개 시 유료 전환 검토

## 8. 향후 확장 고려사항

- **클라우드 이전**: 맥북 의존 탈피 → AWS EC2 t2.micro(무료 1년) 또는 GCP Cloud Run
- **Vercel 크론 연계**: `cron-job.org`로 2시간마다 `/api/cron/verify-albums` 호출

- **Neo4j 엣지 생성 자동화**: 현재 크레딧은 로그에만 기록. 추후 Neo4j에 자동 엣지 생성 연동 예정
