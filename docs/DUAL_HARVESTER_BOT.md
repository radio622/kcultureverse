## 1. 배경: 왜 듀얼봇이 필요한가?

### 기존 데이터 소스의 한계 (2026-03-23 논의에서 실증)

| 소스 | 장점 | 치명적 한계 |
|------|------|-------------|
| **MusicBrainz** | 일부 K-Pop 대형 아이돌 데이터 완벽 | 인디/발라드 아티스트(예: 백아) 앨범 **아예 없음**. 컴필레이션 1개만 등록된 경우 다수 |
| **iTunes API** | 한국 인디까지 99.9% 커버 | 한글 앨범명 대신 **영문 번역명 반환** (예: '우주선' → 'SPACESHIP'). `&lang=ko_KR` 파라미터를 넣어도 영문이 우선 출력됨 |
| **Spotify API** | **완벽한 한글 앨범명** 반환 확인 완료 ('우주선' 그대로 출력) | 발매일이 틀린 경우 다수 (스포티파이 업로드 날짜로 대체되는 현상). **곡별 크레딧(작사/작곡/프로듀서) 미제공** |

### 결론
- 껍데기(앨범명, 커버, 트랙 수) → **Spotify가 최고**
- 발매일 교정 + 크레딧 추출 → **Gemini 3.1이 실시간 웹 검색으로 보완**
- 이 두 개를 결합한 **듀얼 봇(2인 1조)** 구조가 탄생

## 2. 봇 아키텍처

### 전체 작동 흐름도
```
[메인 루프] ─── while True (무한 순환) ───
  │
  ├─ 스케줄러: DB에서 엣지 많은 아티스트 순으로 큐 생성
  │           (이미 수집 완료한 아티스트는 자동 건너뜀)
  │
  ├─ 아티스트 1명 할당
  │
  ├─ 🤖 봇 1 (The Harvester) ── Spotify API
  │   ├─ 아티스트 이름 검색 (1회 호출)
  │   ├─ 앨범 10개씩 페이징 수집 (limit=10)
  │   │   └─ 다음 페이지 있으면 → 32초 쿨타임 후 이어서
  │   ├─ 수집 데이터: 한글 앨범명, 발매일, 커버URL, 타입, 트랙수
  │   ├─ Supabase album_releases에 upsert (중복 방지)
  │   └─ progress.json에 "수집 완료" 체크포인트 기록
  │
  ├─ 🧠 봇 2 (The Curator) ── Gemini 3.1 Flash Lite Preview
  │   ├─ 봇 1이 가져온 앨범 중 미검증 건만 필터링
  │   ├─ 📅 달력 우선순위 정렬:
  │   │   ├─ 1순위: 오늘(월/일)과 같은 발매일 앨범
  │   │   ├─ 2순위: 내일 → 모레 → ... 가까운 날짜 순
  │   │   └─ 3순위: 해당 없는 나머지 미검증 앨범
  │   ├─ 앨범 1건씩 Gemini에게 전달 (google_search 도구 활성화)
  │   ├─ 검증 3대 임무:
  │   │   ├─ ① 최초 한국 실물 발매일 교정
  │   │   ├─ ② 수록곡별 작사/작곡/프로듀싱/피처링 크레딧 추출
  │   │   └─ ③ 아티스트 국적(Korean 여부) 확인
  │   ├─ 검증 후 32초 쿨타임
  │   └─ progress.json에 "검증 완료" 깃발 기록
  │
  ├─ ⏳ 10분 대기 후 다음 아티스트
  │
  └─ 전체 1사이클 완료 시 → 진행률 초기화 → 다시 처음부터 (신규 앨범 체크)
```

### 재처리 방지 이중 잠금 (Double Dedup Guard)

| 방어선 | 방어 대상 | 기록 위치 | 동작 |
|--------|----------|----------|------|
| 🔒 봇 1 잠금 | 아티스트 재수집 | `progress.json → harvested_artists[]` | 이미 수집한 아티스트는 큐에서 제외 |
| 🔒 봇 2 잠금 | 앨범 재검증 | `progress.json → verified_albums[]` | `아티스트명::앨범명` 키로 중복 체크 |
| 🔒 DB 잠금 | 앨범 중복 Insert | `Supabase ON CONFLICT` | 같은 (artist_name, album_title) 조합 자동 무시 |

### 중단 후 이어서 하기 (체크포인트 복원)
- 스크립트 중단 시 `progress.json`이 자동 저장됨
- 재시작 시 파일을 읽고 마지막 처리 지점 다음부터 이어서 작업
- 전체 아티스트 1바퀴 완료 시 `harvested_artists` 초기화 → 재순환 (신규 앨범 감지)

## 3. API 기술 실증 결과 (2026-03-23 테스트 완료)

### Spotify API
- **토큰 발급**: `client_credentials` 방식 정상 작동 확인
- **한글 앨범명**: `백아 → '우주선'`, `아이유 → 'LILAC'` 등 완벽 한글 출력 확인
- **페이지 한도 변경 확인 (잠수함 패치)**: 
  - `limit=50` ❌ `limit=20` ❌ `limit=10` ✅ `limit=5` ✅ `limit=1` ✅
  - **스포티파이가 공식 문서(50개)보다 실제로는 10개로 제한 중** (2026-03 기준)
- **아이유 전체 수집 테스트**: 9페이지 → 74개 앨범 100% 수집 성공 (Rate Limit 0회)
- **API Key**: 기존 `.env.local` 키가 만료(invalid_client)되어 신규 발급 후 교체 완료

### Gemini API
- **모델명 주의**: `gemini-3.1-flash-lite` ❌ (404) → `gemini-3.1-flash-lite-preview` ✅ (정상)
- **공식 문서**: https://ai.google.dev/gemini-api/docs/models/gemini-3.1-flash-lite-preview?hl=ko
- **실시간 웹 검색**: `"tools": [{"google_search": {}}]` 파라미터로 활성화 가능
- **적용 완료 파일 3개**:
  - `src/app/api/cron/verify-albums/route.ts` (Vercel 크론)
  - `src/app/api/universe/suggest/route.ts` (AI 게이트키퍼)
  - `scripts/v7.4-mass-verification.py` (로컬 대량 검증)

### iTunes API
- 기존 코드 5개 파일에서 `&lang=ko_KR` 없이 호출 중이었음 확인
- 한국어 파라미터를 넣어도 영문 우선 반환 → **Spotify로 대체하는 것이 최선**

## 5. 데이터베이스 용량 모니터링 필수 사항

### Supabase (앨범 텍스트 데이터)
- **무료 한도**: 500MB
- **현재 사용량**: ~수 MB (매우 넉넉)
- **예상 증가**: 앨범 10만 건 추가 시 ~20MB. **걱정 불필요**
- **이미지**: 스포티파이 서버 URL을 저장하는 핫링크 방식 → 용량 소모 0 Byte

### Neo4j AuraDB (그래프 엣지/노드)
- **무료 한도**: 관계선(Edges) **최대 175,000개** ⚠️
- **현재 사용량**: ~8,400개 관계선
- **위험 시나리오**: 봇 2가 곡마다 작곡가/프로듀서 선을 무한히 긁어서 그으면 수개월 내 한계 도달 가능
- **대비책**:
  1. 가중치 0.3 이상의 주요 크레딧(메인 작곡/프로듀서/피처링)만 엣지로 생성하도록 가이드라인 강제
  2. 월 1회 `MATCH ()-[r]->() RETURN count(r)` 쿼리로 현재 엣지 수 모니터링
  3. 12만 개 돌파 시 유료 전환(월 $65) 또는 저가중치 엣지 정리 검토

## 7. 향후 확장 고려사항

- **Vercel 크론 연계**: 현재 Vercel 무료 티어는 하루 1회 크론만 허용. `cron-job.org` 같은 외부 무료 스케줄러로 2시간마다 `/api/cron/verify-albums` 호출하면 하루 12회로 증가 가능
- **클라우드 이전**: 맥북 의존을 벗어나려면 AWS EC2 t2.micro(무료 1년) 또는 GCP Cloud Run에 이 스크립트를 올리는 방안 검토
- **Spotify API Rate Limit 고도화**: 현재 32초 고정 쿨타임이지만, 429 응답의 `Retry-After` 헤더를 읽어 동적으로 조절하는 Exponential Backoff 로직이 이미 코드에 탑재되어 있음
