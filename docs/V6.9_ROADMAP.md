## 3. 파이프라인 설계

### 3.1 Step 1: CSV 파싱 및 신규 아티스트 추출

```
[artists/*.csv 18개 파일]
    ↓
[CSV 파싱 → 아티스트 이름 추출]
    ↓
[기존 organic-graph.json과 대조 → 신규 아티스트 필터링]
    ↓
신규 ~250명 리스트
```

### 3.2 Step 2: MusicBrainz MBID 조회 + 한글/영어 이름 결정

- MusicBrainz API로 각 아티스트의 **MBID, 공식 영문명, 한글명** 조회
- 한국 아티스트의 경우 **한글이 대표이름(nameKo = 대표, name = 영문)**
- Rate Limit: **1 req/sec** (차단 방지)
- 조회 실패 시 iTunes Search API로 백업 검색

### 3.3 Step 3: 크레딧 크롤링 (MusicBrainz Recording Credits)

각 신규 아티스트의 **모든 앨범 → 모든 곡의 크레딧** 확인:

```
GET /ws/2/recording?artist={mbid}&limit=100&offset=0&fmt=json&inc=artist-credits
```

크레딧에서 추출할 관계:
- **FEATURED**: 피처링 참여 아티스트
- **PRODUCER**: 프로듀서
- **COMPOSER/LYRICIST**: 작곡가/작사가
- **ARRANGER**: 편곡가

### 3.4 Step 4: 컴필레이션 앨범 크롤링

```
GET /ws/2/release?artist={mbid}&type=compilation&fmt=json&limit=100
```

같은 컴필레이션에 참여한 아티스트끼리 `COMPILATION` 엣지 생성:
- 라벨에 **실제 앨범명** 표기 (예: "No Alternative 참여", "사랑도 통역이 되나요 OST")
- weight: 0.5

### 3.5 Step 5: 우주 재빌드 및 배포

- `v5.4-build-universe.ts` 재실행 → d3-force 좌표 재계산
- `v6.5-fix-korean-names.py` 재실행 → 한글 이름 보정
- `npx next build` → Vercel 배포

## 4. API 차단 방지 전략

| 위험 | 대응 |
|------|------|
| MusicBrainz Rate Limit (1 req/sec) | `await sleep(1100)` 적용. 250명 × 2 API(MBID + 크레딧) = ~10분 |
| MusicBrainz 503 에러 | 3회 Retry with Exponential Backoff (2초 → 4초 → 8초) |
| iTunes API 차단 | 200ms 딜레이 + country=KR 파라미터 고정 |
| 대량 크레딧 크롤링 | 곡 100개 단위 페이징. offset 증분으로 전체 디스코그래피 수집 |

## 5. 한글 이름 결정 기준

| 우선순위 | 기준 | 예시 |
|---------|------|------|
| 1 | CSV에 한글로 기재된 이름 | "잔나비" → nameKo: 잔나비 |
| 2 | MusicBrainz aliases에 한글 alias 존재 | aliases에 "아이유" → nameKo: 아이유 |
| 3 | 영문 공식명이 더 유명한 경우 영문 유지 | "BLACKPINK" → name: BLACKPINK |
| 4 | 한글 이름을 찾을 수 없는 외국인 | "Bob James" → name만 유지 |
